Operating System Fundamentals

What is an Operating System?

An operating system (OS) is system software that manages computer hardware and software resources and provides common services for computer programs.

Functions of an Operating System

  • Process management
  • Memory management
  • File system management
  • I/O device management
  • Security management
  • Networking
  • User interface

Types of Operating Systems

  1. Batch OS: Jobs with similar requirements are batched together and executed
  2. Time-Sharing OS: Multiple users share computing resources simultaneously
  3. Distributed OS: Multiple computers connected over a network appear as a single system
  4. Real-Time OS: Guarantees processing within strict time constraints
  5. Multi-programming OS: Executes multiple programs concurrently

Kernel vs. User Mode

  • Kernel mode: Complete access to hardware and CPU instructions
  • User mode: Limited access to system resources, can perform only what the API allows

Process Management

What is a Process?

A process is a program in execution. Each process has:

  • Program counter
  • Stack
  • Data section
  • Heap
  • Text section (code)

Process States

  • New: Process is being created
  • Ready: Process is waiting to be assigned to a processor
  • Running: Instructions are being executed
  • Waiting: Process is waiting for some event to occur
  • Terminated: Process has finished execution

Process Control Block (PCB)

Contains all information about a process:

  • Process ID
  • Process state
  • Program counter
  • CPU registers
  • Memory management information
  • Accounting information
  • I/O status information

Context Switch

The procedure of saving the state of an active process and loading the saved state of another process.

Scheduling Algorithms

  • First-Come-First-Served (FCFS): Non-preemptive, executes processes in order of arrival
  • Shortest Job First (SJF): Selects the process with the smallest execution time
  • Priority Scheduling: Assigns priority to each process, executes highest priority first
  • Round Robin: Each process gets a small unit of CPU time (time quantum)
  • Multilevel Queue: Processes are placed in different queues based on their priority
  • Multilevel Feedback Queue: Processes can move between queues

Threads

A thread is a lightweight process that shares the same resources as its parent process.

  • User-level threads: Managed by user-level libraries
  • Kernel-level threads: Managed by the operating system

Inter-Process Communication (IPC)

Methods used by processes to communicate with each other:

  • Pipes: Unidirectional data channel
  • Shared Memory: Multiple processes access the same memory region
  • Message Passing: Processes exchange messages through system calls
  • Sockets: Communication endpoints for network communication

Memory Management

Memory Hierarchy

  1. Registers (fastest, smallest)
  2. Cache (L1, L2, L3)
  3. Main Memory (RAM)
  4. Secondary Memory (Disk, SSD)
  5. Tertiary Memory (Tape, optical media)

Memory Allocation Methods

  • Contiguous Allocation: Each process is allocated a single continuous block of memory
  • Non-contiguous Allocation: Process is allocated memory that is not contiguous

Paging

A memory management scheme that eliminates the need for contiguous allocation of physical memory.

  • Page: Fixed-size block of memory
  • Frame: Fixed-size block of physical memory
  • Page Table: Maps logical pages to physical frames

Virtual Memory

A memory management technique that provides an illusion to users of a very large main memory.

Page Replacement Algorithms

  • FIFO (First-In-First-Out): Removes the page that was brought into memory first
  • LRU (Least Recently Used): Removes the page that has not been used for the longest time
  • Optimal: Removes the page that will not be used for the longest time in the future

Thrashing

High paging activity causing the system to spend more time in paging than executing.

File Systems

File Attributes

  • Name
  • Type
  • Location
  • Size
  • Protection
  • Time, date, and user identification

File Operations

  • Create
  • Read
  • Write
  • Reposition
  • Delete
  • Truncate

Directory Structure

  • Single-Level Directory: All files in a single directory
  • Two-Level Directory: Separate directory for each user
  • Tree-Structured Directory: Directories can have subdirectories
  • Acyclic-Graph Directory: Allows sharing of files/directories

File Allocation Methods

  • Contiguous Allocation: Allocates consecutive blocks
  • Linked Allocation: Each file block points to the next block
  • Indexed Allocation: Special block contains pointers to all blocks

Free Space Management

  • Bit Vector: 1 for free block, 0 for allocated block
  • Linked List: Links all free blocks together
  • Grouping: First block in the free block group contains addresses of a few free blocks
  • Counting: Keeps the address of the first free block and the number of contiguous free blocks

Deadlocks

Necessary Conditions for Deadlock

  1. Mutual Exclusion: At least one resource must be held in a non-sharable mode
  2. Hold and Wait: A process must be holding at least one resource and waiting for resources held by other processes
  3. No Preemption: Resources cannot be forcibly taken from a process
  4. Circular Wait: A circular chain of processes, each waiting for a resource held by the next process

Deadlock Handling Strategies

  1. Prevention: Ensure at least one of the necessary conditions cannot hold
  2. Avoidance: Dynamically check if a resource allocation will lead to a deadlock
  3. Detection and Recovery: Allow deadlocks to occur, detect them, and recover
  4. Ignorance: Ignore the problem and assume deadlocks never occur (used in many operating systems)

Banker’s Algorithm

A deadlock avoidance algorithm that checks if a resource allocation will lead to a safe state.

System Design Basics

Key System Design Principles

  1. Scalability: Ability to handle growing amount of work
  2. Reliability: System consistently performs according to specifications
  3. Availability: Time a system remains operational
  4. Efficiency: Optimal use of resources
  5. Maintainability: Ease of maintenance
  6. Security: Protection from unauthorized access

Client-Server Architecture

A computing model where the server hosts, delivers, and manages resources and services requested by the client.

Monolithic vs. Microservices

  • Monolithic: Single-tiered software application
  • Microservices: Collection of loosely coupled, independently deployable services

Load Balancing

Distributes incoming network traffic across multiple servers to ensure no single server is overwhelmed.

Caching

Stores copies of data in a high-speed data storage layer for faster retrieval.

Database Sharding

Partitioning a database across multiple servers to improve performance.

CAP Theorem

States that in a distributed system, you can only have two of:

  • Consistency: All nodes see the same data at the same time
  • Availability: Every request receives a response
  • Partition Tolerance: System continues to operate despite network partitions

System Design Patterns

Architectural Patterns

  • MVC (Model-View-Controller): Separates application into three components
  • Microservices: Collection of loosely coupled services
  • Event-Driven Architecture: Produces, detects, and consumes events
  • Layered Architecture: Organizes system into layers with specific responsibilities

Design Patterns

  • Singleton: Ensures a class has only one instance
  • Factory: Creates objects without specifying the exact class
  • Observer: Defines a one-to-many dependency between objects
  • Strategy: Defines a family of algorithms, encapsulates each one, and makes them interchangeable
  • Adapter: Allows objects with incompatible interfaces to collaborate

Scalability Concepts

Horizontal vs. Vertical Scaling

  • Horizontal Scaling: Adding more machines to the resource pool
  • Vertical Scaling: Adding more power (CPU, RAM) to an existing machine

Database Scaling

  • Master-Slave Replication: Data copied from master to slave databases
  • Master-Master Replication: Data synchronized between multiple master databases
  • Federation: Splitting database by function
  • Sharding: Partitioning data across multiple databases

Caching Strategies

  • Cache-Aside: Application is responsible for reading/writing from storage and cache
  • Read-Through: Cache sits between application and storage, cache provider loads data into cache
  • Write-Through: Data is written to cache and storage simultaneously
  • Write-Behind: Data is written to cache and later synchronized with storage

Message Queues

Enable asynchronous communication between services.

Coding Interview Strategies

Problem-Solving Approach

  1. Understand the problem: Clarify requirements, constraints, and edge cases
  2. Design an algorithm: Start with a brute force approach, then optimize
  3. Implement the solution: Write clean, modular code
  4. Test your solution: Check edge cases and time/space complexity

Data Structures to Master

  • Arrays and Strings
  • Linked Lists
  • Trees and Graphs
  • Hash Tables
  • Stacks and Queues
  • Heaps
  • Tries

Algorithms to Master

  • Sorting (Quicksort, Mergesort, etc.)
  • Binary Search
  • BFS and DFS
  • Dynamic Programming
  • Greedy Algorithms
  • Backtracking

Big O Notation

  • O(1): Constant time
  • O(log n): Logarithmic time
  • O(n): Linear time
  • O(n log n): Log-linear time
  • O(n²): Quadratic time
  • O(2^n): Exponential time

Common Pitfalls

  • Not clarifying the problem
  • Not considering edge cases
  • Not analyzing time and space complexity
  • Not testing the solution
  • Over-engineering the solution

Top Interview Questions & Answers

Operating Systems

Q: What is the difference between process and thread?

A: A process is an independent program in execution with its own memory space, while a thread is a lightweight execution unit within a process that shares the process’s resources (memory space, file descriptors). Multiple threads in a process share the same heap but have their own stack.

Q: What is a deadlock? How can it be prevented?

A: A deadlock is a situation where two or more processes are unable to proceed because each is waiting for resources held by another. Deadlocks can be prevented by:

  1. Mutual Exclusion: Eliminating the concept of mutual exclusion on resources
  2. Hold and Wait: Processes request all resources at once
  3. No Preemption: Allow resources to be preempted
  4. Circular Wait: Impose an ordering on resource requests

Q: Explain virtual memory.

A: Virtual memory is a memory management technique that provides an illusion of a very large main memory. It allows a computer to compensate for physical memory shortages by temporarily transferring data from RAM to disk storage. This enables a program to operate as if it has more memory than the system actually has.

Q: What is thrashing?

A: Thrashing occurs when a system spends more time in page swapping than in executing processes. It happens when the system doesn’t have enough physical memory to handle the required working set sizes of active processes, causing excessive paging operations that severely degrade performance.

Q: Explain the working of LRU page replacement algorithm.

A: Least Recently Used (LRU) algorithm replaces the page that has not been used for the longest period. It works on the principle that pages that have been used recently are likely to be used again. Implementation typically involves a linked list or a counter to track when each page was last accessed.

Q: What is spooling?

A: Spooling (Simultaneous Peripheral Operations On-Line) is a process where data intended for a peripheral device (like a printer) is temporarily held in a buffer (usually disk) until the device is ready to handle it. This allows the CPU to work on other tasks rather than waiting for slower I/O operations to complete.

Q: What is the difference between preemptive and non-preemptive scheduling?

A: In preemptive scheduling, the operating system can interrupt a running process to allocate the CPU to another process based on factors like priority or time quantum. In non-preemptive scheduling, once a process starts execution, it continues until it terminates or blocks for I/O, without being interrupted by the scheduler.

Q: What is a semaphore?

A: A semaphore is a synchronization primitive used to control access to a common resource by multiple processes. It maintains a count that can be incremented by signal operations and decremented by wait operations. Binary semaphores (with values 0 or 1) act like mutexes, while counting semaphores allow multiple processes to access a finite number of resources.

Q: What is internal and external fragmentation?

A:

  • Internal Fragmentation: Occurs when memory is allocated in fixed blocks and the allocated memory is larger than needed, resulting in wasted space within the allocated memory block.
  • External Fragmentation: Occurs when free memory is broken into small blocks over time, and although the total free memory might be sufficient for a process, it cannot be allocated because the memory is not contiguous.

Q: Explain the dining philosophers problem.

A: The dining philosophers problem is a classic synchronization problem where five philosophers sit around a circular table with five forks, one between each pair of philosophers. Each philosopher needs two forks to eat. The challenge is to design a protocol that allows philosophers to eat without deadlocking (where each holds one fork and waits for another) or starving (never getting a chance to eat).

System Design

Q: How would you design a URL shortening service like TinyURL?

A:

  1. API Endpoints: Create endpoints for URL shortening and redirection
  2. Data Storage: Use a database to store mappings between short and long URLs
  3. URL Generation: Either use a hash function or a base-62 conversion of an auto-incremented ID
  4. Rate Limiting: Implement to prevent abuse
  5. Analytics: Track usage statistics
  6. Caching: Use caching to improve performance
  7. Load Balancing: Distribute traffic across multiple servers
  8. Database Sharding: Partition data as the system grows

Q: How would you design a news feed system like Facebook or Twitter?

A:

  1. Data Model: Users, posts, likes, comments, relationships
  2. Feed Generation: Pull model (generate on request) or push model (pre-compute)
  3. Storage: Use a combination of relational and NoSQL databases
  4. Caching: Cache popular posts and user feeds
  5. Load Balancing: Distribute traffic across multiple servers
  6. CDN: Store and deliver media content
  7. Notification System: Notify users of new content
  8. Analytics: Track user engagement

Q: Explain the CAP theorem and its implications for distributed systems.

A: The CAP theorem states that in a distributed system, you can only have two of: Consistency (all nodes see the same data), Availability (every request receives a response), and Partition Tolerance (system continues to operate despite network partitions). In real-world systems, since network partitions are unavoidable, the choice is between CP (consistency and partition tolerance) and AP (availability and partition tolerance) systems, depending on business requirements.

Q: How would you design a distributed cache?

A:

  1. Partitioning: Distribute data across multiple nodes using consistent hashing
  2. Replication: Maintain copies of data for fault tolerance
  3. Consistency Protocol: Ensure data consistency across replicas
  4. Eviction Policy: Implement LRU or other policies for cache eviction
  5. Failure Handling: Handle node failures gracefully
  6. Monitoring: Track cache hit/miss rates and other metrics
  7. Scalability: Allow for easy addition/removal of nodes

Q: How would you design a web crawler?

A:

  1. URL Frontier: Maintain a queue of URLs to be crawled
  2. Fetcher: Download web pages
  3. Parser: Extract links and content from pages
  4. URL Seen?: Check if a URL has been crawled before
  5. URL Filter: Filter out unwanted URLs
  6. Content Store: Store crawled content
  7. Distributed Architecture: Use multiple machines for parallel crawling
  8. Politeness: Respect robots.txt and crawl delays

Q: What factors would you consider when designing a database schema?

A:

  1. Normalization: Reduce redundancy and improve data integrity
  2. Indexing: Optimize for query performance
  3. Relationships: Define appropriate foreign key relationships
  4. Data Types: Choose appropriate types for each field
  5. Scalability: Consider future growth and partitioning needs
  6. Performance: Optimize for common query patterns
  7. Security: Implement proper access controls
  8. Backup and Recovery: Plan for disaster recovery

Q: How would you design a scalable e-commerce website?

A:

  1. User Management: Authentication, profiles, shopping carts
  2. Product Catalog: Categories, search, recommendations
  3. Order Processing: Payment integration, inventory management
  4. Microservices Architecture: Separate services for different functions
  5. Database Design: SQL for transactions, NoSQL for product catalog
  6. Caching: Cache product information, user sessions
  7. CDN: Deliver static content
  8. Load Balancing: Distribute traffic across servers
  9. Scaling Strategy: Plan for horizontal scaling during peak times

Q: How would you design a rate limiter?

A:

  1. Algorithms: Token Bucket, Leaky Bucket, Fixed Window, Sliding Window
  2. Storage: Use Redis or similar for distributed rate limiting
  3. Identification: Define what to rate limit (IP, user ID, API key)
  4. Response: Return appropriate HTTP status codes (429 Too Many Requests)
  5. Headers: Include rate limit information in response headers
  6. Multiple Tiers: Different limits for different user types
  7. Monitoring: Track rate limit violations

Q: Explain sharding and partitioning in databases.

A:
Sharding is a form of horizontal partitioning where a large database is split into smaller, faster, more manageable pieces called shards. Each shard has the same schema but contains a different subset of the data, typically divided by a shard key.

Partitioning can be:

  • Horizontal: Rows are distributed across multiple tables (sharding)
  • Vertical: Columns are distributed across multiple tables
  • Functional: Data is partitioned according to how it is used

Benefits include improved query performance, better scalability, and increased availability, at the cost of more complex application logic and potential issues with joins and transactions across shards.

Q: How would you design a distributed file storage system like Google Drive?

A:

  1. Storage Nodes: Distribute files across multiple storage servers
  2. Metadata Service: Track file locations, permissions, versions
  3. Chunking: Split large files into smaller chunks
  4. Replication: Maintain multiple copies for redundancy
  5. Consistency Protocol: Ensure consistency across replicas
  6. Client Applications: Provide interfaces for various platforms
  7. Synchronization: Keep files in sync across devices
  8. Access Control: Implement sharing and permissions
  9. Version Control: Track file changes
  10. Recovery Mechanisms: Handle node failures and data corruption

Interview Tips

General Tips

  • Understand the basics thoroughly: Deep understanding of fundamentals is more important than shallow knowledge of many topics
  • Practice problem-solving: Use platforms like LeetCode, HackerRank, and CodeSignal
  • Mock interviews: Practice with peers or use interview simulation platforms
  • Explain your thought process: Interviewers are often more interested in how you think than the final answer
  • Learn from rejections: Analyze feedback and improve

Operating Systems Interview Tips

  • Understand core concepts like process management, memory management, and file systems
  • Be able to explain scheduling algorithms and their trade-offs
  • Understand synchronization primitives (mutex, semaphore) and deadlock handling
  • Know about memory allocation techniques and page replacement algorithms
  • Be familiar with I/O management and file system implementations

System Design Interview Tips

  • Follow a structured approach:
    1. Clarify requirements
    2. Define system APIs
    3. Estimate scale and constraints
    4. Design data model
    5. High-level system design
    6. Detailed design for key components
    7. Identify bottlenecks and improvements
  • Start with a simple design and then iterate
  • Consider trade-offs and justify your decisions
  • Be familiar with real-world systems and their design patterns
  • Practice designing popular systems (URL shortener, news feed, chat app)

Coding Interview Tips

  • Start with a brute force solution, then optimize
  • Think aloud – let the interviewer follow your thought process
  • Test your solution with examples, including edge cases
  • Analyze time and space complexity
  • Write clean, modular code with appropriate variable names
  • Ask clarifying questions when needed
  • Practice on a whiteboard or plain text editor

Behavioral Interview Tips

  • Prepare stories that demonstrate your skills and experience
  • Use the STAR method (Situation, Task, Action, Result)
  • Research the company culture and values
  • Prepare questions to ask the interviewer
  • Show enthusiasm and a willingness to learn
  • Be honest about your weaknesses but emphasize how you’re improving
